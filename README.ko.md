코드 리뷰는 "이 코드가 올바른가?"를 묻습니다.  
`skeptic-code`는 묻습니다: **"이 코드가 존재해야 하는가? 그리고 존재해야 한다면 — 현실을 버텨낼 수 있는가?"**

## 왜 기존 도구로는 부족한가

린터와 dead-code 분석기는 공통된 전제를 가집니다: **"실행되면 필요한 것이다."**

`skeptic-code`는 그 전제를 거부합니다. 기존 도구가 잡지 못하는 네 가지 실패 유형:

- **YAGNI 위반** — 실행은 되지만 현재 요구사항과 무관한 코드
- **조용한 실패** — 에러를 삼키는 `except: pass` / `catch (_) {}` 패턴
- **범위 초과** — 아무도 요청하지 않았지만 자연스럽게 추가된 기능
- **검증 없는 가정** — 하드코딩된 값, 무시된 반환값, 누락된 timeout

"유죄 추정" 프레이밍은 같은 코드를 봐도 다른 결론을 만들어냅니다.

## 설치

```bash
/install-skill nuri428/skeptic-code
```

## 사용법

```
/skeptic-code              # 자동 범위 감지
/skeptic-code quick        # 상위 5개 발견, ~2분
/skeptic-code deep         # 전체 라인 수준 감사, ~10분
/skeptic-code <path>       # 특정 파일 또는 디렉토리
```

## 7가지 용의자

| 태그 | 이름 | 죄목 |
|------|------|------|
| `[GHOST]` | 죽은 코드 | 한때 필요했다. 이제는 아니다. 아직도 떠돌고 있다. |
| `[PROPHET]` | 투기적 기능 | 오지 않을 미래를 위해 작성됨. "아마 필요할 것 같아서..." |
| `[LIAR]` | 조용한 실패 | 에러를 처리한다고 주장한다. 실제로는 삼켜버린다. |
| `[TWIN]` | 중복 | 두 곳에 같은 로직. 하나는 존재해서는 안 된다. |
| `[STRANGER]` | 범위 초과 | 아무도 요청하지 않았다. 자연스럽게 추가됐다. 스펙에 없었다. |
| `[ORACLE]` | 검증 없는 가정 | 세상이 협조한다고 전제한다 — 검증도, 대비도, 테스트도 없이. |
| `[CLIFF]` | 무한 실패 경로 | 지금은 동작한다. 한계도, 재시도도, 바닥도 없어서 언젠간 무너진다. |

`[ORACLE]`과 `[CLIFF]`는 기존 5개에 추가된 두 가지입니다. 왜 이 두 개만 추가됐고 무엇이 의도적으로 제외됐는지는 아래 *설계 근거*를 참고하세요.

## 작동 방식

1. **읽기** — 전체 파일, 스키밍 금지
2. **1차 탐색** — 15가지 알려진 패턴으로 후보 식별
3. **2차 검증** — grep으로 실제 사용처 확인 후 `[CUT]` 판정
4. **보고서** — 구체적인 before→after diff 포함한 우선순위 목록
5. **적용** — 사용자 승인 후 수정, 각 컷마다 테스트 실행 확인

**핵심 보장**: grep 증거 없이는 `[CUT]` 없음. 불확실한 발견은 `[QUESTION]` 또는 `[FALSE_ALARM]`이 되고, 조용히 삭제되지 않습니다.

## 범위와 한계

이 스킬이 **커버하지 않는** 것:
- Race condition — 정적 grep으로 신뢰할 수 없음. 이 영역의 false positive는 도구 전체의 신뢰를 망칩니다
- 프로파일링이나 동적 분석이 필요한 런타임 실패 시나리오
- 아키텍처 실패 모드
- 코딩 프로세스 지침 (언제 물어볼지, 어떻게 계획할지)

이런 항목에는 별도의 아키텍처 리뷰 또는 프리모텀 세션을 사용하세요.

## 설계 근거

5개에서 7개 용의자로의 확장은, 이 스킬이 참고한 두 가지 방법론을 확장안 자체에 적용해서 검증했습니다.

**카르파시 "Think Before Coding"을 확장안에 적용:**

확장안이 어떤 가정을 하고 있었는가?

- `[RACE]`는 grep이 race condition을 신뢰할 수 있게 감지한다고 가정했습니다 → **기각**: 동시성 문제의 false positive는 너무 많습니다
- 다중 모드 명령어 구조 (`exist` / `assume` / `fail` / `full`) — Simplicity First 원칙: 요청은 더 넓은 커버리지이지 새로운 명령어 인터페이스가 아님 → **`[PROPHET]`으로 기각**

**확장안에 대한 프리모텀 적용:**

"확장된 스킬이 6개월 후 실패했다 — 왜?"

- `[RACE]` false positive가 개발 시간을 낭비하고 발견에 대한 신뢰를 무너뜨림
- 4개 모드 구조가 사용자를 혼란스럽게 만들어 결국 아무것도 안 씀
- `[ORACLE]` / `[BLIND]` / `[TIGHTROPE]`이 개념적으로 겹침 → `[ORACLE]`로 통합

살아남은 것: grep으로 검증 가능한 패턴을 가진 명확히 구분되는 용의자 두 개. 명령어 인터페이스는 변경하지 않고 기존 Wanted List에 추가.

## 감사의 말

`skeptic-code`는 두 가지 출처에서 형태를 갖췄습니다.

**[Andrej Karpathy의 LLM 코딩 가이드라인](https://x.com/karpathy/status/2015883857489522876)**

*Simplicity First* 원칙 — 요청받지 않은 기능 추가 금지, 단일 사용 코드에 추상화 금지 — 은 `[PROPHET]`, `[STRANGER]`, helper-called-once 체크의 직접적인 출처입니다. *Think Before Coding* — 가정을 명시하고, 구현 전에 확인하라 — 은 `[ORACLE]`의 출처입니다. 시니어 엔지니어가 본능적으로 지키지만 좀처럼 글로 쓰지 않는 것들을 명확하게 정리해주셔서 감사합니다.

**프리모텀 방법론** (Gary Klein / Daniel Kahneman)

실패가 이미 일어났다고 가정하고 거슬러 올라가는 방식은 낙관적 리뷰와 전혀 다른 질의 검토를 강제합니다. 이 스킬의 "유죄 추정" 입장은 프리모텀을 라인 수준에 적용한 것입니다. `[CLIFF]` — 무한 성장, 누락된 timeout, 재시도 없음 — 은 "실제 부하에서 이게 어떻게 실패하는가?"라는 프리모텀 질문을 grep으로 검증 가능한 패턴으로 압축한 것입니다. 프레임워크를 만들어주셔서 감사합니다.

두 접근법 모두 이 스킬이 커버하는 것보다 훨씬 넓습니다. 위의 한계들은 실수가 아니라 의도된 설계 결정입니다.
